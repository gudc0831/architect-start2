# Architect Start 재구축 PLAN

## 방향 판단

이번 재구축은 기존 코드를 복제하는 작업이 아니다.
기존 문서가 보여주는 제품 의도는 유지하되, 개발 방식은 완전히 다시 짠다.

이번 프로젝트의 고정 전제는 아래와 같다.

- 개발효율이 최우선이다
- 개발은 로컬 중심으로 진행한다
- Firebase는 앱 플랫폼이 아니라 DB 용도로만 사용한다
- GitHub는 여러 환경에서 이어 작업하기 위한 코드 동기화 수단으로 사용한다
- 배포는 후순위다. 지금은 최소 기반만 열어둔다
- 로그인은 핵심 화면이 모두 나온 뒤에 붙인다
- 초기 개발 단계에서는 모든 페이지를 열어둔다

## 이번 PLAN의 핵심 결론

이 프로젝트는 `local-first 개발 구조 + Firestore DB + 로컬 파일 저장 + 나중에 확장 가능한 auth 뼈대`로 시작하는 것이 맞다.

즉, 지금은 아래를 목표로 삼는다.

- 화면과 도메인 흐름을 빠르게 만든다
- DB 연결은 유지하되 인증/배포 때문에 개발이 막히지 않게 한다
- 모든 페이지를 로컬에서 바로 들어가보며 작업할 수 있게 한다
- 이후 로그인, 권한, 배포를 덧붙일 수 있는 구조만 남겨둔다

## 서비스 정의

Architect Start는 공동주택 설계 프로젝트에서 발생하는 업무, 도면/자료, 변경이력, 운영 상태를 하나의 흐름으로 관리하는 협업 플랫폼이다.

### 초기 타깃

- 공동주택 프로젝트를 수행하는 3~20명 규모 설계팀
- 회사 전체 도입이 아니라 프로젝트팀 단위 도입
- 엑셀, 메신저, 메일, 폴더에 흩어진 업무와 도면 대응 흐름 정리

### 초기 제품 가치

- 업무와 파일을 연결한다
- 엑셀형 작업 흐름을 유지한다
- 실무 도메인 필드를 처음부터 반영한다
- 나중에 권한/로그인/배포를 붙여도 구조가 흔들리지 않게 만든다

## 이번 재구축에서 유지할 것 / 버릴 것

### 유지할 것

- 프로젝트 -> 이슈/작업 -> 파일 -> 휴지통 흐름
- 공동주택 설계 실무 특화 방향
- 엑셀형 보드 UX
- 파일 연결 중심 작업 방식
- 작은 팀 기준의 빠른 개발 문화

### 버릴 것

- 배포 기준으로 먼저 구조를 짜는 방식
- Firebase Hosting/Auth 전제를 깔고 시작하는 방식
- route handler 안에 로직이 몰리는 구조
- 로그인 완성 후에야 화면을 볼 수 있는 흐름
- 기능 구현 전에 너무 큰 인프라 결정을 잠그는 방식
- 초기부터 workspace를 넣는 복잡한 멀티 구조

## 단일 구조 원칙

초기 버전은 `workspace` 없이 `project` 하나만 최상위 엔티티로 둔다.

### 이유

- 지금 목표는 로컬에서 전체 흐름을 빠르게 확인하는 것이다
- 멀티 워크스페이스 구조는 초기에 개발 복잡도만 높인다
- 나중에 필요하면 `workspace`는 상위 래퍼로 추가할 수 있다

### 초기 최상위 구조

- project
- issue/task
- file
- trash

## 기술 방향

### 기본 선택

- Frontend/App: Next.js
- Language: TypeScript
- DB: Firebase Firestore만 사용
- 파일 저장: 로컬 폴더에 복사 업로드
- Firestore에는 메타데이터만 저장
- auth, hosting, scheduler는 지금 잠그지 않는다
- 구조는 service/use-case/repository 분리로 시작한다

### Firebase를 어떻게 쓸 것인가

Firebase는 지금 단계에서 `백엔드 플랫폼`이 아니라 `DB 공급자`로 다룬다.

즉 아래처럼 이해한다.

- 사용한다: Firestore 데이터 저장
- 지금은 보류한다: Firebase Auth
- 지금은 보류한다: Firebase Hosting/App Hosting
- 지금은 보류한다: Firebase Storage
- 지금은 보류한다: 운영 자동화 스택

### 왜 이 방향이 맞는가

- 로그인/배포 때문에 개발이 막히지 않는다
- 로컬에서 대부분의 페이지를 바로 확인할 수 있다
- 여러 인터넷 환경에서도 작업 이어가기가 쉽다
- 나중에 인증을 붙여도 구조를 깨지 않고 확장 가능하다

## 개발 구조 원칙

### 1. local-first

- 기본 개발 환경은 항상 로컬이다
- 앱은 로컬에서 바로 뜨고, 핵심 페이지를 즉시 들어가볼 수 있어야 한다
- 인터넷이 불안정해도 UI와 도메인 흐름 작업이 가능해야 한다

### 2. DB와 앱 로직 분리

- Firestore 호출은 repository 계층으로 격리한다
- 화면은 repository를 직접 호출하지 않는다
- 서비스 계층에서 비즈니스 규칙을 모은다
- 나중에 DB를 바꾸더라도 화면 수정이 작게 끝나게 만든다

### 3. 로그인은 최소 뼈대만

지금 로그인 완성에 시간을 쓰지 않는다.
대신 나중에 붙일 수 있도록 아래 정도만 준비한다.

- `auth interface` 정의
- `getCurrentUser()` 같은 추상 함수 준비
- `AuthProvider` 자리만 만든다
- dev 모드에서는 `guest/local user`로 모든 페이지 접근 가능하게 둔다
- 실제 보호 라우트는 후속 작업으로 분리한다

### 4. 페이지 진입 우선

초기에는 아래 원칙으로 간다.

- 주요 페이지는 로그인 없이 로컬 진입 가능
- 라우트 보호는 넣지 않는다
- fake current user만 주입한다
- 실제 auth gate를 붙일 위치만 명확히 남겨둔다

### 5. 권한은 문서로만 선잠금

권한 모델은 문서로는 먼저 잠그되, 구현은 2단계로 나눈다.

- 1단계: 화면/도메인/데이터 흐름 먼저 구성
- 2단계: 중앙 권한 서비스와 보호 라우트 적용

이렇게 해야 개발이 덜 막힌다.

## 파일 처리 원칙

### 초기 파일 방식

- 파일은 로컬에서 선택한다
- 앱이 지정된 로컬 프로젝트 폴더로 파일을 복사 저장한다
- Firestore에는 파일 메타데이터만 저장한다
- 파일은 할 일 하나에 여러 개 연결 가능하게 한다

### 초기 저장 방식

- 실제 파일: 프로젝트 루트 바깥의 별도 로컬 데이터 폴더 저장
- DB: Firestore 메타데이터 저장
- 연결 구조: `task 1 : N files`

### 이유

- 경로 링크 방식은 기기마다 깨질 가능성이 높다
- 복사 업로드가 사용성과 일관성 면에서 낫다
- 이후 Firebase Storage 또는 다른 스토리지로 바꿀 때도 메타데이터 구조를 재사용할 수 있다

## 핵심 페이지 구조

초기 1차 페이지는 아래 4개로 고정한다.

1. 보드
2. 작업목록(일별) - 스프레드시트형
3. 작업목록(월별) - 달력형
4. 휴지통

### 페이지 관계

- 보드와 작업목록(일별)은 같은 task 데이터를 다른 방식으로 보여주는 핵심 화면이다
- 각 task에는 관련 파일을 여러 개 연결할 수 있어야 한다
- 월별 화면은 `기한일` 기준으로 일정을 보여준다
- 휴지통은 task, file, 연결정보 전체를 소프트삭제 기준으로 다룬다

## 일별 작업목록 시트 원칙

이미지 기준의 시트 구조를 1차 기준으로 고정한다.
시트가 뒤섞이지 않도록 컬럼 의미를 명확히 유지한다.

### 1차 고정 컬럼

- 작업번호
- 기한/날짜
- 분류
- 요청자
- 담당자
- 항목
- 작성일시
- 일일등록
- 설명
- 진행상황
- 진행사항
- 결론
- 관련파일

### 시트 규칙

- 컬럼 순서는 쉽게 바꾸지 않는다
- 각 컬럼의 의미를 혼합하지 않는다
- 상세 메모는 `설명`, 진행 로그는 `진행사항`, 최종 정리는 `결론`으로 분리한다
- `관련파일`은 셀 하나에 요약 표시하고 실제 연결 목록은 패널 또는 모달에서 관리한다

### 기본 동작 규칙

- 작업번호는 `MIL-662` 같은 형식으로 자동 발급한다
- 일별 시트 기본 정렬은 `기한일 오름차순`으로 둔다
- 월별 달력은 `기한일` 기준으로 표시한다
- 휴지통 기본 정렬은 `삭제일 내림차순`으로 둔다

## 월별 달력 원칙

- 달력은 `기한일` 기준으로 표시한다
- task 제목과 상태가 한눈에 보이도록 최소 표현으로 시작한다
- 세부 수정은 달력 화면이 아니라 상세 패널에서 처리한다

## 휴지통 원칙

휴지통은 아래 범위를 모두 포함한다.

- task
- file 메타데이터
- task-file 연결정보

### 삭제 정책

- 초기에는 소프트삭제만 적용
- 복구 가능 상태 유지
- 영구삭제는 후속 단계로 미룬다

### 공통 엔티티

## 로그인 전략

현재 단계의 로그인 전략은 `완성이 아니라 확장 포인트 확보`다.

### 지금 할 것

- auth 관련 폴더 구조 생성
- 현재 사용자 타입 정의
- guest/local session mock 제공
- 향후 real auth를 연결할 entry point 마련

### 지금 하지 않을 것

- Firebase Auth 연동 완료
- OAuth 세부 플로우
- 세션 만료 처리
- 권한 기반 리다이렉트 완성

### 이유

지금 로그인부터 완성하면, 페이지 확인과 UI 작업이 느려지고 전체 개발효율이 떨어진다.
초기엔 `로그인 없는 개발 가능 상태`가 더 중요하다.

## GitHub 작업 방식

여러 환경에서 이어 작업할 것이므로, Git 전략은 단순해야 한다.

### 권장 전략

- `main`: 현재 기준선
- `feat/*`: 짧은 기능 브랜치
- `design/*`: 짧은 디자인 실험 브랜치
- `fix/*`: 수정 브랜치
- `develop` 같은 장수 브랜치는 두지 않는다

### 판단

디자인을 따로 브랜치하는 것은 가능하다.
하지만 오래 유지하면 구조 브랜치와 충돌이 커진다.
따라서 `design/*`는 짧게 쓰고 빠르게 병합하거나 버리는 방식이 맞다.

### 작업 원칙

- 하루 안에 끝나는 작은 단위로 자른다
- 페이지 단위보다 흐름 단위로 자른다
- 예: `shell`, `board-read`, `task-panel`, `file-list`, `auth-stub`

### 로컬 환경 공통화

- `.env.example` 제공
- Firebase Firestore 연결값만 최소 포함
- 설치 후 바로 실행 가능한 README 유지
- 기기가 바뀌어도 `install -> env -> dev` 순서만으로 시작 가능해야 한다

## 데이터 입력 원칙

- 로컬 개발용 샘플 데이터는 미리 대량으로 넣지 않는다
- 사용자가 직접 입력하면서 테스트할 수 있게 구조를 만든다
- seed는 비워둔 상태로 시작해도 되게 만든다

## 디자인 원칙

- 초기 디자인은 최소한으로 간다
- 레이아웃과 정보 분리, 시트 구조, 패널 구조가 우선이다
- 시트별 의미가 뒤섞이지 않게 데이터 배치를 먼저 고정한다
- 시각적 완성도는 후속 지시에서 보강한다

## 추천 폴더 구조

```text
src/
  app/
  domains/
    task/
    file/
    project/
    auth/
  use-cases/
  repositories/
    firestore/
    memory/
  providers/
  lib/
  mocks/
  types/
```

### 구조 의도

- `repositories/firestore`: 실제 DB 접근
- `repositories/memory`: 로컬 개발용 대체 저장소
- `domains/auth`: 나중에 로그인 붙일 자리
- `mocks`: 페이지 진입용 최소 fake user 정도만 유지

## 이번 PLAN의 최종 판단

- 개발효율이 우선이므로 local-first가 맞다
- Firebase는 지금 Firestore DB 용도로만 쓰는 것이 맞다
- 배포/운영은 지금 과하게 설계하지 않는다
- 로그인은 최소 확장 포인트만 만들고 뒤로 미룬다
- 초기에는 모든 페이지를 바로 열어볼 수 있어야 한다
- project 단일 구조로 시작하는 것이 맞다
- 파일은 로컬 복사 업로드 + Firestore 메타데이터 저장이 맞다
- 달력은 기한일 기준으로 본다
- 휴지통은 task, file, 연결정보 전체를 다룬다
- Git은 `main + 짧은 feat/* + 짧은 design/*` 전략이 맞다
- 업로드 파일은 프로젝트 루트 바깥 별도 데이터 폴더에 저장한다
- 보드와 일별 작업목록은 같은 task 데이터의 다른 뷰로 다룬다
- 작업번호는 자동 발급으로 간다
- 파일 버전은 초기엔 최소 구조만 열어두고 후속 확장한다
- 기본 정렬은 일별 `기한일 오름차순`, 휴지통 `삭제일 내림차순`으로 고정한다
